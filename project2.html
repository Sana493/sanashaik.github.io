When the Most Important Data Disappears

Can we still predict diabetes without its strongest signal? Testing machine learning robustness under real-world data scarcity.

The Question That Started This

Diabetes has always been a part of my life — not just something I learned about in a classroom, but something I've watched up close in my own family. The daily routines, the careful meals, the glucose checks. And through all of it, one thing stood out: everything seemed to hinge on a single number. Blood sugar.

That observation became a question I couldn't shake: what if that number wasn't available? In real healthcare settings, data is missing all the time. A patient might not have had a recent glucose test. Records get lost. Tests don't get run. So, if a prediction model completely depends on glucose to work, is it actually useful in the real world?

Research Question: When the most important clinical feature — glucose — is removed, can machine learning models still make meaningful predictions about diabetes risk? And if so, what features step in to fill the gap?

The Dataset

The Pima Indians Diabetes Dataset (sourced from Kaggle / UCI ML Repository) contains 768 patient records from Pima Indian women, each with 8 clinical measurements and a binary outcome: diabetic or not.

  • Pregnancies — number of pregnancies
  • Glucose — plasma glucose concentration (2-hour oral glucose tolerance test)
  • Blood Pressure — diastolic blood pressure (mm Hg)
  • Skin Thickness — triceps skinfold thickness (mm)
  • Insulin — 2-hour serum insulin (μU/mL)
  • BMI — body mass index (kg/m²)
  • Diabetes Pedigree Function — a score reflecting family history
  • Age — in years

While exploring, several features contained zero values where zero is medically impossible. These were placeholders for missing data and required imputation before modeling.

Exploratory Data Analysis

Fig 1. Distribution of all 9 variables after preprocessing. Insulin and SkinThickness are strongly right-skewed. The Outcome histogram reveals class imbalance: ~500 non-diabetic vs 268 diabetic cases (~65/35 split) — a model that always guesses "no diabetes" would still achieve 65% accuracy.

Fig 2. Correlation heatmap. Glucose has the strongest correlation with Outcome at 0.49 — nearly double BMI (0.31) and Age (0.24). This confirmed the premise: glucose dominates, and its removal will meaningfully change how the model reasons.

EDA Insight: The correlation heatmap doesn't just show what predicts diabetes — it reveals the structure of substitution. When glucose is removed, the model will need to lean on BMI, Age, and Diabetes Pedigree Function. We can see this coming before running a single experiment.

Data Preprocessing

Biologically impossible zeros in Glucose, Blood Pressure, Skin Thickness, Insulin, and BMI were treated as missing values and replaced with the column median from valid (non-zero) entries.

Why median, not mean? Features like Insulin are heavily right-skewed with extreme outliers. The mean would be pulled upward, producing imputed values that don't represent the typical patient. The median is resistant to that distortion.

  • 80% training / 20% testing with stratification to preserve the 65/35 class balance
  • StandardScaler applied for Logistic Regression — fit on training data only to prevent leakage
  • Tree-based models used raw unscaled values (inherently scale-invariant)
  • Random seed fixed at 42 for reproducibility
Acknowledged limitation: Median imputation introduces bias, especially for Insulin which has an unusually high rate of zeroed values. KNN imputation could provide more realistic estimates in future work by using relationships between features rather than a single population value.

Models & Why I Chose Them

  • Logistic Regression — The interpretable linear baseline. Sensitive to feature scale. If this performs well, the problem is linearly separable; if it struggles, non-linear patterns matter.
  • Decision Tree — Captures non-linear relationships by splitting the feature space into regions. Easy to interpret, but prone to overfitting without depth constraints.
  • Random Forest (200 trees) — An ensemble that builds many trees on random subsets, then averages predictions. More robust than a single tree and provides built-in feature importance scores — ideal for understanding what the model relies on when key features disappear.

Experiments

Four feature configurations were tested across all three models, progressively stripping away predictive power to observe how each model degrades — and what it falls back on.

🧪 Experiment 1 — All Features

All 8 variables included. Sets the performance ceiling and confirms feature dominance before any removals.

🧪 Experiment 2 — Remove Glucose

The single most important feature is dropped. The core question: what's left when the dominant signal disappears?

🧪 Experiment 3 — Remove Glucose, BMI & Age

The three highest-correlated features are gone. Only moderate and weak signals remain.

🧪 Experiment 4 — Weak Features Only

Only the five lowest-correlated variables remain. Tests the absolute floor of predictive power.

Results

Fig 3. Accuracy across all four experiments. The key story is the shape of the decline — gradual, not sudden. Random Forest consistently leads and never falls below ~67%. Logistic Regression shows the sharpest drop from Exp 1 to Exp 2.

What this graph actually proves: Removing glucose hurts — but it doesn't break the model. Random Forest dropped roughly 5 percentage points, then stabilized. This gradual degradation, rather than collapse, is the key finding. It means these models have genuine flexibility — they aren't entirely dependent on a single feature.
Logistic Regression
Strong with all features. Most sensitive to glucose removal — the linear boundary relies heavily on that single dimension. Drops noticeably in Exp 2.
Decision Tree
Moderate and inconsistent. Performance fluctuates across experiments, likely due to overfitting on the small dataset. Least reliable of the three.
Random Forest ✦
Best overall. Most robust to feature removal. Leverages diverse tree structures to find patterns even in weak feature sets. Recommended model.

The Real Cost of Missing Data

Accuracy tells you how often you're right. In healthcare, how you're wrong matters just as much. The confusion matrices below show where Logistic Regression fails across each experiment.

Fig 4a. Exp 1 — All Features. 82 true negatives, 27 true positives. 18 false negatives.

Fig 4b. Exp 2 — No Glucose. 80 true negatives, 25 true positives. 29 false negatives.

Fig 4c. Exp 3 — No Glu/BMI/Age. 85 true negatives, 21 true positives. 33 false negatives.

Fig 4d. Exp 4 — Weak Only. 85 true negatives, 21 true positives. 33 false negatives.

The pattern that matters most: As features are removed, the model doesn't just become less accurate — it specifically becomes worse at detecting diabetic patients. False negatives climb from 18 to 33. A false negative means a diabetic patient is told they're fine. That's not a statistical error — it's a real health risk.

Feature Importance

Random Forest reveals its reasoning through feature importance scores — how much each variable reduces prediction error, averaged across all 200 trees.

Fig 5. With all features, Glucose accounts for ~27% of decision-making weight — nearly double BMI's contribution (~15%). The dominance is unmistakable.

Fig 6. Without Glucose, BMI rises to ~21%, followed by DiabetesPedigreeFunction and Age. The model doesn't collapse — it redistributes weight to the next best signals.

Why the redistribution matters: The model isn't guessing randomly when glucose is gone — it pivots to features capturing different dimensions of risk. BMI reflects metabolic health. Diabetes Pedigree Function captures inherited risk. Age reflects cumulative physiological exposure. Together, they tell a different story — but still a meaningful one.

Limitations, Ethics & Reflection

  • Population specificity: This dataset represents only Pima Indian women. Diabetes risk factors differ significantly across ethnicities, genders, and regions. Do not apply this model to broader populations without revalidation.
  • Imputation assumptions: Replacing missing values with medians is a reasonable first step — but it's an assumption, not a measurement. Insulin imputation is especially uncertain.
  • Class imbalance not corrected: The 65/35 class split was not addressed with resampling or class weighting. This likely suppresses recall for the diabetic class.
  • No hyperparameter tuning: The Decision Tree had no depth constraint, making overfitting likely.
  • Ethical use: These models are research tools — not diagnostic instruments. Predicting "likely has diabetes" should never replace a clinical evaluation.
Personal reflection: The part of this project that stayed with me wasn't the accuracy numbers — it was the false negatives. Watching that number climb from 18 to 33 as features were removed made it concrete. Behind each of those 33 predictions is a person who has diabetes and would be told they don't. That's what makes getting this right matter.

Conclusion

This project started as a prediction task and became something more meaningful. By deliberately removing the most important feature, I was able to see how models adapt, what they fall back on, and precisely where and how they start to fail.

The core finding: even when the most important data disappears, patterns remain. The model doesn't know glucose is gone; it just works with what it has. And in this dataset, what's left is enough to still say something real about diabetes risk.

For me, this connects back to what I observed growing up. Diabetes isn't just one number. It's a combination of family history, body composition, age, and physiology building over time. The data reflects that complexity — and so, to its credit, does the model.
